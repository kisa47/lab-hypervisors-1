# Лабораторная работа по теме "Типы и функции гипервизоров"

## Требования к выполнению работы

- Компьютер с операционной системой Debian и установленным гипервизором KVM
- Образ диска виртальной noble-server-cloudimg-amd64.img

## Цель работы

- Научиться запускать виртуальные машины в гипервизоре KVM с использованием Qemu с заданными параметрами через терминал.
- Проверить на практике возможность запуска ОС в виртуальной машине отличной от ОС хаста.
- Проверить на практике работу механизмов изоляции и абстрагирования ресурсов.

## Подготовка к проведению практической работы 

Проверьте наличие файла диска виртуальной машины командой:
```bash
ls  ~/images/noble-server-cloudimg-amd64.img
```

Если файла нет, то необходимо его скачать командой:
```bash
wget -O ~/images/noble-server-cloudimg-amd64.img http://cloud-images.ubuntu.com/noble/current/noble-server-cloudimg-amd64.img
```

## Теоретическая часть

Гипервизор (hypervisor) является ключевым компонентом облачных технологий и виртуализации, играющим важную роль в современном мире информационных технологий. Основная задача гипервизора состоит в предоставлении и координации ресурсов виртуальным машинам таким образом, чтобы они могли функционировать параллельно и независимо (изолированно) друг от друга и от системы хоста, сохраняя эффективность и надежность всей системы.

KVM (Kernel-based Virtual Machine) — программное решение для аппаратной виртуализации в операционных системах Linux и её дистрибутивах. Это модуль ядра Linux, который превращает ОС в гипервизор первого типа (Bare Metal) — то есть работающий сразу поверх железа без лишних уровней абстракции.

QEMU (Quick Emulator) — программа с открытым исходным кодом для эмуляции аппаратного обеспечения различных платформ. Позволяет запускать виртуальные машины с различными операционными системами на одном физическом устройстве.

## Проведение лабораторной работы

### Подготовка образа диска виртуальной машины

Задать пароль для учетной записи суперпользователя **root** командой, где **PASSWORD** - пароль:
```bash
virt-customize -a  ~/images/noble-server-cloudimg-amd64.img --root-password password:PASSWORD
```

### Старт виртуальной машины

запустить виртуальную машину со следующими параметрами:
- cpu host: Использует возможности процессора хоста для ускорения работы виртуальной машины
- type=q35: Машинный тип q35 обеспечивает поддержку современного оборудования, совместимого с ACPI
- accel=kvm: Включает ускорение KVM (Kernel-based Virtual Machine) для повышения производительности виртуализации
- объем выделенной ОЗУ 2Gb
- с отключенным графическим интерфесом ВМ (вывод идёт непосредственно в консоль)
- в режиме снимка (не изменяя исходный образ диска)
- с переадресацией сетевых соединений с порта 2222 хоста на 22 порт виртуальной машины

```bash
qemu-system-x86_64  \
  -cpu host -machine type=q35,accel=kvm -m 2048 \
  -nographic \
  -snapshot \
  -netdev id=net00,type=user \
  -device virtio-net-pci,netdev=net00 \
  -drive if=virtio,format=qcow2,file=/home/b415/images/noble-server-cloudimg-amd64.img
```
### Настройка виртуальной машины

После запуска ВМ необходимо дождаться завершения загрузки ОС (может занять несколько минут). Признаком того, что ОС виртуальной машины запустилась будет наличие в выводе строки ```Ubuntu 24.04.4 LTS ubuntu ttyS0```. Далее необходимо выполнить вход в ОС виртуальной машины с учетными данными суперпользователя **root** и заданным на этапе **Задание пароля пользователя root для виртуальной машины** паролем.



Для настройки сети в ОС виртуальной машины сначала нужно получить название сетевого интерфейса командой ```ip link```.

Далее необходимо создать файл /etc/netplan/50-cloud-init.yaml следующего содержания, где **enp0s2** - сетевой интерфейс виртуальной машины:
```yaml
network:
  version: 2
  renderer: networkd
  ethernets:
    enp0s2: 
      dhcp4: true
      dhcp6: false
```
И применить конфигурацию командой ```netplan apply```. После применения конфигурации проверить доступ в сеть Интернет выполнив ```ping vshte.ru```

Установить утилиту для проверки и нагрузочного тестирования ОЗУ **memtester** командой:

```bash
add-apt-repository universe # добавить universe репозиторий 
apt-get update && apt-get install -y stress-ng
```

### Получение данных в виртуальной машины.

После успешного входа необходимо получить следующие данные:

- версию ядра ОС и архитектуру виртуальной машины командой ```uname -a```
- Объем оперативной памяти ВМ командой ```free -h```
- модель процессора командой ```lscpu```
- получить древо процессов командой ```pstree -Aptl```
- список сетевых интерфесов (кроме lo) командой ```ip link```
- список pci устройств командой ```lspci```

### Полученние данных с ОС хоста

Не выходя из ВМ и не завершая её работу открыть другую вкладку в терминале, собрать в нём аналогичные данные и вписать их в раздел **Полученные результаты**.

### Нагрузочное тестирование и поломка ВМ

Выполнить в Виртуальной машине команду для нагрузочного тестирования ОЗУ:
```bash
stress-ng --vm 2 --vm-bytes 100% --vm-method all --verify -t 10m
```

Перейти в терминал хоста и пронаблюдать как изменилось потребление оперативной памяти. Записать результат команды во время нагрузочного тестирования ```free -h``` в раздел **Полученные результаты**.

Так как сейчас работает в виртуальной машине как суперпользователь, перед тем как мы сломаем ОС виртуальной машины можно попробовать повыполнять любые действия (даже приводящие к фатальным последствиям).

Далее необходимо выполнить имитацию последствий хакерской атаки направленной на удаление данных. Для это попробуем начать полную перезапись данных диска ВМ.

Получим устройство которое нужно перезаписать "нулями" командой:
```bash
df / | tail -n 1 | awk '{print $1}'
```

Само устройсто не будет иметь дисковой метки тома (цифры), например вывод команды */dev/vda1*, значит нужное нам устройство */dev/vda*.

И выполним запись всего диска "нулями":
```bash
dd if=/dev/zero of=/dev/vda bs=1M status=progress
```

После завершения попробовать разные команды. Посмотреть как система работает.

Переключиться в терминал хоста и посмотреть, не затронули действия по уничтожению данных на работающей ВМ работсопособность узла.

Записать полученные наблюдения в раздеое **Полученные результаты**

### Завершение работы ОС виртуальной машины.

Так как система сломана и выключиться стандартными командами больше не может необходимо перейдите в вкладку терминала с виртуальной машиной и нажать комбинацию клавишь Ctrl+A и следом сразу X. Главное удостовериться, что выбрана английская раскладка.

Так как мы запускали виртуальную машину в режима "снимка" можно выполнить повторный запуск виртуальной машины и убедится, что в течении нескольких минут может быть восстановлена работа виртуальной машины. Снимки создают (переключают рабочую виртуальную машину в режим снимка) для возможного отката к состоянию при измении конфигурации или в качестве резервной копии. При работе ВМ в режиме снимка не происходит изменение файла диска, поэтому его можно скопировать и отключить режим снимка.
